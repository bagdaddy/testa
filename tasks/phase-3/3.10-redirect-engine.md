## Goal

The state-of-the-art redirect engine. Bar is VWO/ABTasty parity-or-better. Specifics in the architecture doc and grilling memories above. This task is large; if it grows past the PR-size cap, split into 3.10a (decide+execute), 3.10b (cross-domain), 3.10c (SPA path).

## Acceptance criteria

### `decide.ts`
- `decide(experiment, currentUrl, assignment, sessionState): RedirectDecision | null` returns the target URL or null.
- Reads the variation's `redirect` change (`{ type: 'redirect', from_url, to_url }`).
- Honors URL match types: `exact`, `contains`, `notContains`, `regex`, `startsWith`, `endsWith`.
- POST-back guard: skip redirect if `document.referrer` matches the source URL pattern within `cooldown_ms` (default 30 s).
- Returns the destination URL preserving query params (UTM + custom) and hash unless the variation explicitly overrides.

### `execute.ts`
- `execute(decision: RedirectDecision): void` performs the redirect.
- Default: `location.replace(target)` (no back-button reentry).
- For SPA same-app same-origin redirects: `history.replaceState(null, '', target)` plus dispatch a synthetic `_testa:locationchange` to trigger re-eval, then return without full pageload.
- Sets `Analytica.isRedirecting = true` for the duration.
- Logs decision + outcome to `__pixel_debug.redirects` ring buffer.

### `loop-guard.ts`
- Session-scoped `Set<experimentId>` of "already redirected this session".
- Prevents ping-pong if the destination URL also matches a rule.
- Stored in `sessionStorage` under `_testa_redirected_in_session` so it survives same-tab reloads but not new tabs.

### `cross-domain.ts`
- When destination is a different eTLD+1, append `?_tu=<short-encoded-visitor-id>` to the destination URL.
- On the destination side, the worker reads `_tu` from the inbound request, sets the `_testa_uuid` cookie if not already present, then `history.replaceState`s to strip `_tu` from the address bar.
- "Different eTLD+1" detection uses Public Suffix List — but for v1, a simpler `host !== source.host` check is acceptable; document the simplification.

### `spa-path.ts`
- For SPA app-internal redirects (Next.js, React Router), prefer `history.replaceState` to keep the SPA hydrated.
- Detection: same-origin AND same first path segment AND `Analytica.isNextApp || window.__REACT_ROUTER` heuristic.
- Cross-app or cross-origin: fall back to `location.replace`.

### Tests
- Each module has unit tests under happy-dom.
- The Next.js race condition fix is asserted: redirect decision is computed AFTER framework's hydration microtask (assert via `queueMicrotask` ordering with a mocked Next.js router patch).
- POST-back guard prevents form-submission redirect kills (referrer check).
- Loop guard blocks second redirect within session.
- Cross-domain `_tu` round-trip via mock fetch.

## Implementation notes

Per the grilling memory: ALL eight monkey-patch failure modes from 3.6 must be defended against. The "compute decision in microtask after framework's hydration" pattern is critical — never decide redirects synchronously inside the patched `pushState`.

Diagnostic breadcrumb log streams every redirect decision + observed landing URL into `__pixel_debug` for forensics — required by the memory `known_pixel_spa_bug.md`.

## Out of scope

- The harness that proves SPA correctness across Next.js / react-router-dom (3.11).
- Anti-flicker — customer SmartCode handles.
