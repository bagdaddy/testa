## Goal

Pixel-side evaluator for the new `AudienceCondition` JSON shape. Tier 1 + Tier 2 dimensions. Sandboxed custom JS expression compiler. Legacy `targeting[]` evaluator for 3.3.x/3.6 projects unchanged.

## Acceptance criteria

### Type definition (`packages/shared-types/src/audience.ts`)
Mirror the schema in `docs/reference/audience-schema.md`. Discriminated union by `fact`. Exhaustive types catch missing dimensions in the evaluator at compile time.

### `audience.ts` evaluator
- `evaluate(condition: AudienceCondition, ctx: Context): boolean` walks the tree.
- `Context` carries `page` (url, queryParam map, referrer), `visitor` (uuid, isReturning, cookies, dataLayer accessor), `geo` (country, region), `device` (type, browser, os, language, viewportWidth), `time` (now), `experiments` (assignment lookup).
- `all` → every child true.
- `any` → at least one true.
- `not` → child false.
- Each leaf evaluates per the op (e.g. `'in'`, `'contains'`, `'regex'`, `'between'`).
- Unknown facts fail closed (return `false`) and emit a `_pixel_health` debug event.

### `custom-js.ts` sandbox
- Accepts the AST emitted by crobot at config-publish time (NOT raw JS strings; the AST is part of the audience JSON).
- Evaluator walks the AST with a fixed context (`visitor`, `page`, `session`).
- Allowed nodes: `Literal`, `Identifier`, `MemberExpression` (with bounded depth), `BinaryExpression` (`===`,`!==`,`<`,`<=`,`>`,`>=`,`+`,`-`,`*`,`/`,`%`), `LogicalExpression` (`&&`,`||`), `UnaryExpression` (`!`), `ConditionalExpression` (`?:`), `CallExpression` against a whitelisted set of method names (`includes`, `startsWith`, `endsWith`, `test` on regex literals, `String`, `Number`, `Boolean`, `Array.isArray`, `Date.now`).
- Disallowed: any `=` assignment, `Function`, `eval`, `import`, `fetch`, `window`, `document`, anything outside the whitelist. Reject with descriptive error at compile time (in crobot) AND at evaluator load time (defense in depth).

### `legacy.ts` evaluator
- For projects on `integration_version` `3.4` or `3.6`, ignored — the worker serves the frozen 3.x bundle. But for 4.0 projects whose admin still uses the old `targeting[]` form (transition period), this evaluator handles it.
- Mirrors 3.3.3's "OR-within-type, AND-across-types" group semantics from `crobot/.../3.3.3/script.js`.
- Once admin UI migrates to the new shape (crobot side), this evaluator can deprecate.

### Tests
- Truth tables for each leaf op against synthetic contexts.
- Nested `all`/`any`/`not` produce the right boolean for ~10 hand-built scenarios from `audience-schema.md` examples.
- Custom JS sandbox: `Number(visitor.dataLayer.order_count) > 3` evaluates correctly; rejected ASTs include any window access, any `eval`, any assignment.
- Legacy targeting[]: same fixtures as 3.3.3's tests if available; otherwise hand-built.

## Implementation notes

For `device.type` detection, reuse `is_mobile()` / `is_tablet()` regex patterns from 3.3.3 to preserve compat. Document that the regex is the legacy approach and that future improvements need backwards-compat tests.

For `time.window` op `between`: parse `from` and `to` as ISO 8601 with the supplied `tz`. Use `Intl.DateTimeFormat({ timeZone: tz })` — no extra deps.

For `visitor.dataLayer`: read from `window.dataLayer` if present (GTM convention); each leaf's `path` field is dot-separated (`'user.tier'`).

For `visitor.isReturning`: derive from cookie age — if `_testa_uuid` cookie's age (set timestamp encoded in cookie or computed via a separate `_testa_first_seen` cookie set on first ever visit) > 24h, returning.

## Out of scope

- The crobot-side AST compiler (lives in crobot, runs at config publish time).
- Rule-builder admin UI (crobot's `ProjectResource`).
- Edge-side evaluation (CNAME-edge premium offering, future).
