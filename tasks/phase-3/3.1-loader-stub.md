## Goal

The synchronous loader script that customers' SmartCode embeds in `<head>`. Sets up the `window._testa` queue so customer code can call `_testa.track(...)` / `_testa.consent(...)` immediately, even before the runtime bundle finishes loading. Installs the SPA history monkey-patch at the earliest possible moment.

The loader is the *only* sync `<script>` we serve. Everything else (the runtime) loads with `defer`.

## Context

- The edge worker (Phase 2.7) serves `GET /projects/:slug.js` — that response inlines this loader plus a `<script defer src="…runtime.js?h=…">` tag.
- The customer's SmartCode (provided as a copy-paste snippet, version-controlled separately) hides body until `_testa.load()` resolves; the loader does NOT inject any visual shield.
- The loader bundle is ~5 KB after esbuild minify. Stay tight.

## Acceptance criteria

- `apps/pixel/src/loader.ts` exists. ESM, no async/await at top level, no large polyfills.
- After load, `window._testa` is a function-with-properties: calling `_testa.track('foo', {bar:1})` queues into `window._testa.q` if the runtime hasn't hydrated, or forwards to the live API if it has.
- `_testa.q` is a JS Array. Runtime hydrates by reading + replaying it.
- All public methods (`track`, `trackPurchase`, `consent`, `identify`, `navigate`, `load`) exist as queued stubs from the moment loader runs.
- `_testa.load()` returns a Promise that resolves once runtime fires `_testa._loaded()`. Customer SmartCode awaits this.
- History `pushState` / `replaceState` are monkey-patched idempotently (`window._testa_patched_v4` guard, see `monkey-patch.ts`). Patch dispatches `_testa:locationchange` as a microtask after the original returns.
- `popstate` and `hashchange` listeners installed.
- `pageshow` listener re-installs the patch after bfcache restore.
- Unit tests in `__tests__/` cover: queue receives + drains, monkey-patch is idempotent across double-load, microtask ordering, bfcache re-install, and same-URL pushState filter.

## Implementation notes

```ts
// apps/pixel/src/loader.ts (sketch)
import { installQueue } from './loader/queue.ts';
import { installMonkeyPatch } from './loader/monkey-patch.ts';

if (!(window as any)._testa_patched_v4) {
  installQueue();
  installMonkeyPatch();
  (window as any)._testa_patched_v4 = true;
}
```

Queue stubs return `void`; they don't crash if runtime never loads (e.g. blocked by ad-blocker — queue stays in memory, harmless).

Patch design (verbatim from `01-tracker.md` § SPA navigation):
1. Capture native references early.
2. Install via `Object.defineProperty` (non-writable) so subsequent framework patches wrap us, not the other way around.
3. Wrapper calls original first, then queues a microtask (`Promise.resolve().then`) that fires `_testa:locationchange` CustomEvent.
4. Idempotent guard against double-install.
5. `pageshow` listener re-checks the guard.
6. Compare canonical URL (sorted query keys, strip `_testa_*` params) on each event — same-URL `pushState` is dropped.

`_testa.load()`:

```ts
let loadResolve: () => void;
const loadPromise = new Promise<void>((r) => { loadResolve = r; });
_testa.load = () => loadPromise;
_testa._loaded = () => loadResolve();   // runtime calls this once experiments resolved
```

The runtime overrides `_testa.track` etc. with the live impls during hydration; the `q` array is drained in arrival order.

## Tests

vitest + happy-dom. Cover:
- `_testa.track('e', {})` before runtime loads → present in `_testa.q` as `['track', 'e', {}]`.
- Runtime hydration replaces `_testa.track` with live impl; subsequent calls don't re-queue.
- Loading the script twice is idempotent (no double-patched history).
- `pushState(state, '', '/foo')` triggers a `_testa:locationchange` CustomEvent, scheduled as microtask (verified via `queueMicrotask` ordering).
- `pushState(state, '', location.href)` (same canonical URL) does NOT fire the event.
- After a synthetic `pageshow` event, the patch is still in place.
