## Goal

Continue calling crobot's existing `/api/leads`, `/api/leads/convert`, `/api/pixel` endpoints exactly as 3.6 does, so MySQL-backed dashboards keep working through the cutover.

## Acceptance criteria

- `postLead(experiment, variationId, visitorId)` POSTs `/api/leads` with the legacy body shape (verify against 3.6 source).
- `postLeadConvert(experimentId, goalId, data)` POSTs `/api/leads/convert`.
- `postPixel(...)` GET `/api/pixel?...` for Shopify Custom Pixel events.
- Calls fire from the same lifecycle points as 3.6:
  - Lead on `'variation_applied'`.
  - Convert on goal trigger.
  - Pixel forward on every Shopify event message.
- Endpoints read from `Analytica.domain` (set from `cfPrefill.apiUrl ?? window.apiUrl`).
- Dedup: don't re-fire `/api/leads` for the same `(experimentId, visitorId)` within the same session — reuse 3.6's `Analytica.sent[experimentId]` flag.
- Use `fetch keepalive` so calls survive page unload.
- Tests verify body shape, dedup, and lifecycle wiring.

## Implementation notes

Read `crobot/.../3.6/script.js` for the exact body shapes. Mirror byte-for-byte. The `LeadController` on the crobot side validates strictly.

```ts
// apps/pixel/src/runtime/legacy-http.ts (sketch)
export async function postLead(
  experimentId: number,
  variationId: number,
  visitorId: string,
): Promise<void> {
  if (window.Analytica.sent[experimentId]) return;
  await fetch(`${window.Analytica.domain}/api/leads`, {
    method: 'POST',
    keepalive: true,
    headers: window.Analytica.headers,
    body: JSON.stringify({
      experiment: experimentId,
      variation: variationId,
      visitor: visitorId,
      // ... whatever 3.6 sends; verify
    }),
  });
  window.Analytica.sent[experimentId] = 1;
}
```

## Tests

- Lead body matches 3.6's body for the same input (snapshot test).
- Convert body matches.
- Dedup: second call with same experimentId is a no-op.
- Pixel forward URL contains the right query params.
