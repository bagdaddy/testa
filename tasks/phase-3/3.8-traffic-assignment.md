## Goal

Deterministic variation assignment. Same visitor + same experiment → same variation, every time, across pageloads, across devices (when visitor_id is shared). 3.6 visitors with cached assignments in `_testa_exp_<id>` cookies keep their assignment via cookie-first lookup; only brand-new visitors get bucketed via the new hash.

## Why deterministic, not Math.random()

The user's rationale 2026-05-06: "that pseudo random stuff causes slight SRM in random cases which is annoying and confusing for mid specialists." `Math.random()` produces small but real Sample Ratio Mismatch even at large samples — splits like 50.4 / 49.6 instead of exact 50/50. Deterministic bucketing eliminates this drift to the precision of the hash mod operation.

Bonus reasons:
- **Cross-device consistency** — same visitor signed in across devices (with `_testa.identify()` propagating visitor_id) gets the same variation everywhere.
- **QA forcing** — set `_testa_uuid` to `qa_test_v1` and you reproducibly get a specific variation. Customers' QA teams can test specific paths.
- **Audit** — "why did this visitor see variation B?" → `xxhash32('uuid_abc:17') % 100 = 73, falls in [50, 99] → variation B`.

## Acceptance criteria

- `bucketOf(visitorId: string, experimentId: number): number` is pure, returns 0–99, uses `xxhash32(visitorId + ':' + experimentId, seed=0xABCDEF)`.
- `assign(experiment, visitorId, cachedAssignment): { variationId, isExcluded }` is pure and deterministic.
- Bucket maps onto cumulative `weight` ranges (variations sorted in stable order by `variation_id`).
- `traffic_allocation < 100` → bucket >= traffic_allocation means visitor is excluded.
- **Cookie-first lookup**: if `_testa_exp_<experimentId>` already has a cached variation that still exists in `experiment.variations`, return it without rehashing.
- If the cached variation no longer exists (admin removed it), fall back to fresh hash + overwrite cookie.
- xxhash32 implementation lives in `apps/pixel/src/utils/xxhash.ts` (~30 LOC, sync, no deps).
- The hash seed `0xABCDEF` is a **frozen constant**; never change it (would re-bucket every visitor without a cookie). Document this in `xxhash.ts`.

```ts
// apps/pixel/src/runtime/experiments/traffic.ts (sketch)
import { xxhash32 } from '../../utils/xxhash.ts';

const HASH_SEED = 0xABCDEF;

export function bucketOf(visitorId: string, experimentId: number): number {
  return xxhash32(`${visitorId}:${experimentId}`, HASH_SEED) % 100;
}

export function assign(
  experiment: Experiment,
  visitorId: string,
  cachedAssignment: number | null,
): { variationId: number; isExcluded: boolean } {
  if (
    cachedAssignment !== null &&
    experiment.variations.some((v) => v.variation_id === cachedAssignment)
  ) {
    return { variationId: cachedAssignment, isExcluded: false };
  }
  const bucket = bucketOf(visitorId, experiment.experiment_id);
  if (bucket >= experiment.traffic_allocation) {
    return { variationId: -1, isExcluded: true };
  }
  // Map (bucket - excluded_offset) onto cumulative variation weights.
  const variation = pickByWeight(
    experiment.variations,
    bucket - (100 - experiment.traffic_allocation),
  );
  return { variationId: variation.variation_id, isExcluded: false };
}
```

## Tests

- Determinism: same `(visitorId, experimentId)` produces the same `variationId` 100/100 times.
- Distribution: with `traffic_allocation=100` and 50/50 weights over 10,000 synthetic visitor IDs, splits to within 3σ of 5000/5000 (~ ±150).
- xxhash32: matches a reference vector from the public spec for at least 5 known inputs.
- Cached assignment preferred over fresh hash even if variations list reorders.
- Cached assignment that no longer exists in `experiment.variations` triggers re-bucketing.
- Excluded visitors (bucket >= traffic_allocation) get `isExcluded: true`, `variationId: -1`.

## Backwards-compat note

3.6 used `Math.random()` (verified by reading `crobot/resources/js/integration/3.6/script.js`). 3.6 visitors with `_testa_exp_<id>` cookies keep their assignment via the cookie-first lookup — they are never re-bucketed by the new hash. Only fresh visitors (no cookie) get the new deterministic bucket. The cutover is non-disruptive.
