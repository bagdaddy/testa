## Goal

Deterministic variation assignment. Same visitor + same experiment → same variation, every time, across pageloads. Pre-existing 3.6 visitors must keep their assignment after the cutover (no mid-experiment shuffling).

## Acceptance criteria

- `assign(experimentId: number, variations: Variation[], visitorId: string): variationId` is pure and deterministic.
- Hash function: `xxhash32` (or murmur3 — pick one and document) of `${visitorId}:${experimentId}` → bucket 0–99.
- Bucket maps onto cumulative `weight` ranges (variations summed in stable order by `variation_id`).
- `traffic_allocation < 100` → bucketing first decides include/exclude; included visitors then bucketed across variations.
- Visitor's previous assignment cached in `_testa_exp_<experimentId>` cookie. On subsequent pageviews, the cached value is returned without rehashing — even if the variation list changed.
- If the cached variation no longer exists (admin removed it), fall back to a fresh hash and overwrite the cookie.
- **Backwards-compat constraint:** must produce the same bucket as 3.6 for the same `(visitorId, experimentId)` pair when the same hash function is used. Capture 3.6's hash function from `crobot/.../3.6/script.js` and either reuse it verbatim or prove equivalence with property tests on a corpus of 10k pairs.

## Implementation notes

Look at how 3.6 buckets:

```sh
grep -nE "rand|hash|bucket" ~/projects/crobot/resources/js/integration/3.6/script.js
```

If 3.6 uses `Math.random()` (non-deterministic, persisted in cookie), our new code must keep behaving the same: assign once, cache in cookie, reuse forever. The "consistent-hash" terminology applies to first-assign decisions for new visitors only — old visitors keep their cookie value.

```ts
// apps/pixel/src/runtime/experiments/traffic.ts (sketch)
export function assign(
  experiment: Experiment,
  visitorId: string,
  cachedAssignment: number | null,
): { variationId: number; isExcluded: boolean } {
  if (cachedAssignment !== null && experiment.variations.some((v) => v.variation_id === cachedAssignment)) {
    return { variationId: cachedAssignment, isExcluded: false };
  }
  const bucket = bucketOf(visitorId, experiment.experiment_id);   // 0..99
  if (bucket >= experiment.traffic_allocation) {
    return { variationId: -1, isExcluded: true };
  }
  // map bucket onto cumulative weights
  const variation = pickByWeight(experiment.variations, bucket - (100 - experiment.traffic_allocation));
  return { variationId: variation.variation_id, isExcluded: false };
}
```

## Tests

- Determinism: same `(visitorId, experimentId)` produces the same `variationId` 100/100 times.
- Distribution: with `traffic_allocation=100` and 50/50 weights, 10000 random visitor IDs distribute to 5000±100 each side.
- Cached assignment is preferred over fresh hash even if variations list reorders.
- Cached assignment that no longer exists in `experiment.variations` triggers re-bucketing.
- Excluded visitors (bucket >= traffic_allocation) get `isExcluded: true` and `variationId: -1`.
- 3.6 parity test (only if we can extract a deterministic hash from 3.6; if 3.6 uses `Math.random()`, this test is skipped and documented).
