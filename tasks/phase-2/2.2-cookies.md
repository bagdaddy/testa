## Goal

A single `cookies.ts` module that:

1. Reads `_testa_uuid` from incoming `Cookie` header. If absent, generates a fresh UUID v4.
2. Decides cookie scope: if request `Host` matches a known `tracking_domain` from KV → `Domain=.{customer-domain}`; else → `Domain=.{COOKIE_FALLBACK_DOMAIN}`.
3. Builds a `Set-Cookie` header to refresh the cookie (`Max-Age=63072000`, `SameSite=Lax`, `Secure`, `Path=/`).
4. Special-cases denied consent: `Max-Age=0` to evict + return a daily-rotating ephemeral visitor_id (handled in Phase 2.3 enrich; this module exposes the helper).

## Acceptance criteria

- `getOrCreateVisitorId(request, env)` returns `{ visitor_id, set_cookie_header }`.
- `evictUuidCookie(domain)` returns a `Set-Cookie` string that deletes the cookie.
- `domainForHost(host, kvProjectConfig)` returns the right `Domain=` value or the fallback.
- All functions pure-ish (no global state); easy to unit-test.
- Cookie attributes match exactly: `Domain=...; Max-Age=63072000; Path=/; SameSite=Lax; Secure; HttpOnly` (revisit `HttpOnly`: probably NO because the runtime needs to read it — confirm against `runtime/cookies.ts` plan in Phase 3.3, then pick. v1 ships **without** HttpOnly so JS can read).
- 95%+ coverage on the module; cover both shared-domain and CNAME paths.

## Implementation notes

```ts
// apps/edge/src/cookies.ts (sketch)
const UUID_COOKIE = '_testa_uuid';
const TWO_YEARS_S = 63_072_000;

export interface VisitorIdResult {
  visitor_id: string;
  is_new: boolean;
  set_cookie_header: string;
}

export async function getOrCreateVisitorId(
  request: Request,
  env: Env,
): Promise<VisitorIdResult> {
  const cookies = parseCookies(request.headers.get('cookie') ?? '');
  let visitor_id = cookies[UUID_COOKIE];
  let is_new = false;
  if (!visitor_id) {
    visitor_id = crypto.randomUUID();
    is_new = true;
  }
  const host = new URL(request.url).hostname;
  const domain = await domainForHost(host, env);
  const set_cookie_header =
    `${UUID_COOKIE}=${visitor_id}; Domain=${domain}; Max-Age=${TWO_YEARS_S};` +
    ` Path=/; SameSite=Lax; Secure`;
  return { visitor_id, is_new, set_cookie_header };
}
```

For `domainForHost`: the worker needs to know the project's `tracking_domain` to decide. Two options:

- **Cheap**: maintain a KV reverse-index `host:track.example.com` → `project_slug` so we don't iterate all configs. Add this when Phase 5.4 publishes config from crobot.
- **v1**: derive from request — if `host` starts with `track.` and the parent domain is in a small KV `customer_hosts` set, use it; else fall back. Simpler.

For Phase 2.2, ship the function with a `host` → `domain` rule that:

- Hostname `track.testa.com` (or any `*.workers.dev`) → fallback `Domain=.testa.com`.
- Hostname `track.<x>` where `<x>` exists as a KV key under `customer_hosts:<x>` → `Domain=.<x>`.
- Anything else → fallback.

The KV index `customer_hosts:<domain>` is populated by Phase 5.4 (`PublishProjectConfigToKV` job in crobot). v1 worker tolerates an empty index and just uses the fallback domain.

## Tests

`apps/edge/src/__tests__/cookies.test.ts`:

- New visitor (no cookie): generates UUID, returns `set_cookie_header` matching the spec.
- Existing visitor: returns the same UUID, set-cookie refreshes Max-Age.
- CNAME host (`track.example.com` with KV entry): Domain=.example.com.
- Shared host (`track.testa.com`): Domain=.testa.com.
- Unknown CNAME (no KV entry): falls back to .testa.com.
