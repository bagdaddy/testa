## Goal

A Bun-runnable CLI that applies CH migrations from `apps/collector/db/migrations/` in filename order, tracking applied migrations in CH's `_migrations` table to avoid re-applying.

## Context

The `_migrations` table itself is created by `009_create_migrations_tracker.sql`. The runner must handle the bootstrap: if `_migrations` doesn't exist, apply migrations 001…009 unconditionally; from then on, skip files already in `_migrations`.

`@clickhouse/client` is the official Node-compatible driver and works under Bun. Use it directly (don't go through the singleton from task 1.3 — keep `migrate.ts` self-contained so the singleton's startup order isn't a concern).

## Acceptance criteria

- `bun run migrate` applies all pending migrations in order.
- Re-running `bun run migrate` is a no-op (logs "no pending migrations").
- A failure in one file aborts; subsequent files are NOT applied; the failing file is NOT marked applied.
- Logs are clear: which file is being applied, success/failure, total count.
- Reads CH config from `src/config.ts`.
- Migration files are read as plain text and split by `;` only if needed; pass to CH via `client.command({ query })`.

## Implementation notes

```ts
// apps/collector/src/db/migrate.ts (sketch)
import { createClient } from '@clickhouse/client';
import { readFile, readdir } from 'node:fs/promises';
import { join } from 'node:path';
import { config } from '../config.ts';

const MIGRATIONS_DIR = join(import.meta.dir, '..', '..', 'db', 'migrations');

async function main(): Promise<void> {
  const client = createClient({
    url: config.clickhouse.url,
    username: config.clickhouse.user,
    password: config.clickhouse.password,
    database: config.clickhouse.database,
  });

  const files = (await readdir(MIGRATIONS_DIR)).filter((f) => f.endsWith('.sql')).sort();

  // Find applied set. _migrations may not exist yet on first run.
  let applied = new Set<string>();
  try {
    const result = await client.query({ query: 'SELECT filename FROM _migrations', format: 'JSONEachRow' });
    const rows = (await result.json()) as { filename: string }[];
    applied = new Set(rows.map((r) => r.filename));
  } catch {
    // table doesn't exist yet — first run, fine.
  }

  let count = 0;
  for (const filename of files) {
    if (applied.has(filename)) continue;
    const sql = await readFile(join(MIGRATIONS_DIR, filename), 'utf8');
    console.log(`[migrate] applying ${filename}...`);
    await client.command({ query: sql });
    await client.command({
      query: 'INSERT INTO _migrations (filename) VALUES ({filename:String})',
      query_params: { filename },
    });
    count++;
  }

  console.log(`[migrate] done — ${count} new migration(s) applied`);
  await client.close();
}

main().catch((err) => {
  console.error('[migrate] failed:', err);
  process.exit(1);
});
```

Note: `client.command` runs single-statement queries. CH 24+ supports semicolon-separated batches via the HTTP interface; `client.command` should accept the whole file as one query. If a single file has multiple statements, set `clickhouse_settings: { allow_multi_statement: 1 }` or split on `;` only when outside string/comment.

For v1 keep each migration file to a SINGLE statement to avoid the multi-statement complexity. The schema in `1.1` is already structured this way (one CREATE per file).

Manual verification:

1. `docker compose down clickhouse && docker volume rm tp-clickhouse-data && docker compose up -d clickhouse`
2. `bun run migrate` — applies 9 migrations.
3. `bun run migrate` — no-op (`done — 0 new migration(s) applied`).
4. `clickhouse-client -q "SELECT * FROM _migrations ORDER BY applied_at"` — 9 rows.

## Out of scope

- Down-migrations / rollback. We don't support them in v1.
- Dry-run mode.
- Concurrent migration runs (use a single migration runner per env; advisory lock not needed).
- Programmatic invocation from app startup (always run as a CLI step).

## Done looks like

`bun run migrate` is a clean idempotent CLI step you can include in any deploy script. Twice-running is safe.
