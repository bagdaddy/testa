## Goal

Sync daily FX rates from a free public API (Frankfurter) and expose them at `GET /_internal/fx-rates` so ClickHouse's `fx_rates` dictionary can pull them. Allows query-time conversion of `value_native` to `report_currency`.

## Acceptance criteria

- A scheduled task pulls rates from `https://api.frankfurter.app/latest` (or a configured `FX_API_URL`) once per day.
- Rates are persisted in CH itself (a small `fx_rates_history` table), keyed by date + ccy pair.
- `GET /_internal/fx-rates` returns the most recent N days (default 90) as `JSONEachRow` for the CH dictionary.
- ClickHouse dictionary `fx_rates` (created in migration 008) reads from this endpoint and refreshes every 24 h.
- Idempotent: re-running the sync for the same day overwrites the row, doesn't duplicate.
- Logs new ccy pairs added.

### `fx/store.ts` — DB layer

A new CH table (NOT a materialized view; just storage). Add a 010 migration:

```sql
-- apps/collector/db/migrations/010_create_fx_rates_history.sql
CREATE TABLE IF NOT EXISTS fx_rates_history (
    date     Date,
    from_ccy LowCardinality(String),
    to_ccy   LowCardinality(String),
    rate     Float64
)
ENGINE = ReplacingMergeTree
PARTITION BY toYYYYMM(date)
ORDER BY (date, from_ccy, to_ccy);
```

### `fx/sync.ts` — Frankfurter pull

Frankfurter response (https://api.frankfurter.app/latest?from=USD):

```json
{ "amount": 1, "base": "USD", "date": "2026-05-05",
  "rates": { "EUR": 0.924, "GBP": 0.793, "JPY": 154.21, ... } }
```

Strategy: pull rates with USD as base; for any from_ccy → to_ccy conversion, the dashboard does `(rate_to_USD_for_from_ccy)^-1 * rate_to_USD_for_to_ccy`. Simpler than pulling all-pairs.

Or — pull `latest?from=USD&to=EUR,GBP,...` for an explicit list of customer-relevant currencies, and similarly with each as base. v1 ships USD-base only and computes cross-rates at query time.

```ts
import { upsertRates, type FxRow } from './store.ts';
import { config } from '../config.ts';

export async function syncToday(): Promise<number> {
  const url = `${config.fxApiUrl}/latest?from=USD`;
  const res = await fetch(url);
  if (!res.ok) throw new Error(`Frankfurter ${res.status}`);
  const body = (await res.json()) as { date: string; rates: Record<string, number> };

  const rows: FxRow[] = Object.entries(body.rates).map(([to, rate]) => ({
    date: body.date,
    from_ccy: 'USD',
    to_ccy: to,
    rate,
  }));
  // Add identity (USD→USD = 1) so queries don't special-case
  rows.push({ date: body.date, from_ccy: 'USD', to_ccy: 'USD', rate: 1 });

  await upsertRates(rows);
  return rows.length;
}
```

### `fx/route.ts` — Hono route

```ts
import { Hono } from 'hono';
import { recentRates } from './store.ts';

export const fxRoutes = new Hono();

fxRoutes.get('/_internal/fx-rates', async (c) => {
  const days = Number(c.req.query('days') ?? 90);
  const rows = await recentRates(days);
  // CH dictionary expects JSONEachRow with `date` as DateTime
  const body = rows
    .map((r) => JSON.stringify({
      date: `${r.date} 00:00:00`,
      from_ccy: r.from_ccy,
      to_ccy: r.to_ccy,
      rate: r.rate,
    }))
    .join('\n');
  return c.body(body, 200, { 'content-type': 'application/x-ndjson' });
});
```

### Schedule

For v1, sync runs:

- **At server startup** (one call to `syncToday()` after a 30-second delay so CH connection is stable).
- **Daily** via a `setInterval` inside the HTTP server process.

Production schedule via `cron` or a k8s CronJob is a deploy concern, not a code concern.

```ts
// apps/collector/src/index.ts (additions)
import { syncToday } from './fx/sync.ts';

setTimeout(() => {
  syncToday().catch((e) => console.error('[fx] initial sync failed', e));
}, 30_000);

setInterval(() => {
  syncToday().catch((e) => console.error('[fx] daily sync failed', e));
}, 24 * 60 * 60 * 1000);
```

## Out of scope

- Multi-source FX (only Frankfurter). Add resilience later.
- Real-time rates (daily is plenty for AOV reporting).
- Currency aliases (e.g. CNH vs CNY).
- Historical backfill at startup (we ship empty; rates accumulate from day 1).
